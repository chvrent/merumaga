# mail-magazine-maker 開発・保守総括 (2026/05/15)

本ドキュメントは、プロジェクト「mail-magazine-maker」のこれまでの修正経緯、現在の仕様、および今後の開発・保守における絶対的な注意事項をまとめたものです。

## 1. プロジェクト概要
GASによるWebアプリを用いたメルマガ配信マスタ管理ツール。週次サイクル（火曜始まり）での配信枠管理と、PRコンテンツの紐付け管理を主機能とする。

## 2. 実装済み重要機能と仕様

### 2.1 配信カレンダー (Calendar 2.0)
- **週判定ロジック**: 2026/04/28(火)を基準日とした「火曜〜月曜」サイクルを採用。
- **ドラッグ＆ドロップ (DnD)**: カレンダー上のカードを移動可能。移動はマスターデータ(`app_schedule`)を直接更新する方式。
- **過去データ制限**: 当日より14日以上前のデータは編集不可（グレーアウト）。
- **ヘッダー表示**: 各セルのヘッダー（内容・通数・設定・確認）は日付の下に集約し、シンプル化。

### 2.2 PR管理 (PR 2.0)
- **コンテンツ主体管理**: PRマスタ(`app_pr`)とメルマガの紐付けを `app_pr_targets` シートで行う。
- **紐付け UI**: PR編集モーダルにて、アクティブなメルマガを検索・複数選択（チェックボックス）するUIを実装。
- **データ表示**: 一覧テーブルに「紐付け先」列を追加し、メルマガ名をカンマ区切りで表示。

## 3. 絶対ルール (GEMINI.md 継承)

### 3.1 UI/UX の制約
- **構造の厳守**: Flexbox によるカラム構造を維持すること。
- **非同期UIの徹底**: `google.script.run` 使用時は、必ずボタンの無効化（`disabled`）とローディング表示を行うこと。

### 3.2 データ操作・保守の制約
- **Header-based Property Mapping**: 全てのデータ操作において、スプレッドシートの「ヘッダー名（1行目）」をプロパティキーとして使用すること。独自のエイリアス名称（`linked_newsletters` 等）の使用やハードコーディングを禁止する。
- **バルク処理の徹底**: シートへの書き込みは `setValues` を基本とし、ループ内での `setValue` / `appendRow` は厳禁。

## 4. 保守上の注意事項・トラブルシューティング

- **SyntaxError (識別子重複/構文崩れ)**: `Client.html` 編集時の文字化けや括弧の対応崩れが頻発する可能性がある。`replace` 使用時は、影響範囲を可能な限り小さくし、必ず `grep` 等で実コードを確認してから置換すること。
- **シートの依存関係**:
    - `app_schedule_archives`: 定期ロック機能で必須。決して削除しないこと。
    - `app_pr_targets`: PRとメルマガの紐付けテーブル。保存処理時には必ずこのシートを最新に保つこと。
- **配信編集モーダル**: メルマガ名は編集不可・クリックコピー専用。編集項目は `app_schedule` ヘッダーに合わせ、日付×メルマガ単位の差分は `app_check_status` の `occurrence_override` に保存すること。
- **設定者Rルール**: 設定者が `R` の場合、確認者は空欄のまま選択不可にすること。
- **確定バックアップ**: カレンダーの `設定` と `確認` が同じ日付・同じメルマガで両方赤くなった時点で、`app_schedule_archives` に単日バックアップを作る。`occurrence_override` がある場合は差分適用後の内容を保存する。
- **自動求人系の表示名**: `job_url` は「自動求人特集URL」、`auto_job_feature_id` は「自動求人特集ID」に統一すること。
- **デプロイ運用**: デプロイ後は必ず `運用台帳.md` にバージョン（@番号）と変更内容を明記すること。
